# AGENTS

## 套件版本安全規則（防供應鏈投毒）

安裝或新增任何 npm / Python 套件時，必須遵守以下規則：

### 版本選擇
- **禁止使用發布未滿 7 天的版本**。選擇版本時，優先使用已發布至少一週、且下載量穩定的版本
- 如果最新版本發布不到 7 天，應退回使用上一個穩定版本
- 可透過 `npm view <package> time` 或 `pip index versions <package>` 查詢發布時間

### 版本鎖定
- `package.json` 中使用**精確版本**（如 `"axios": "1.14.0"`），避免使用 `^` 或 `~` 範圍
- Python 專案在 `requirements.txt` 中使用 `==` 鎖定版本
- 確保 lockfile（`package-lock.json`、`pnpm-lock.yaml`、`yarn.lock`）始終存在且被 commit 進 repo

### 可疑套件檢查
- 安裝前檢查套件的 npm/PyPI 頁面，注意：發布時間異常、維護者變更、新增不明依賴
- 如果發現可疑跡象，主動告知使用者並暫停安裝

## 專案定位

本專案不預設特定技術棧。只有在需求明確時，才新增 Node、Python、React 或其他框架相關檔案。

根目錄只放專案入口文件、專案級設定與必要的管理檔案。不要把輸出檔、暫存檔、測試草稿或一次性報告直接放在根目錄。

## 專案結構與檔案位置規則

- `config/`：放非機密、可提交的設定檔。
- `docs/`：放文件、需求、規格與說明。
- `output/`：放程式或 agent 產生的輸出，預設不提交。
- `src/`：放正式原始碼。
- `tmp/`：放暫存檔，預設不提交。
- `.env`：放在專案根目錄，僅供本機使用，不提交。
- `.env.example`：放在專案根目錄，用來說明本專案需要的環境變數名稱與格式，可以提交。

不確定檔案應放在哪裡時，先詢問使用者，不要自行新增不明用途的資料夾。

## 機密與 Git 提交規則

- 不得提交 `.env`、密鑰、token、密碼、私人憑證或個人敏感資料。
- 不得將機密資訊寫進程式碼、文件、進度紀錄或輸出檔。
- `.env.example` 是環境變數名稱、格式與用途說明來源；只能包含環境變數名稱、格式說明、用途說明、合理範例值或空值，不得包含真實機密。
- 新增、刪除或修改 `.env` 的環境變數時，必須同步更新 `.env.example`，確保 key、格式與必要註解一致，但不得填入真實機密。
- `.env` 與 `.env.example` 的非敏感預設值應盡量保持一致；只有密鑰、token、密碼、私人憑證、本機路徑或其他敏感 / 本機專用值可以不同。
- `output/`、`tmp/`、依賴資料夾與快取檔預設不提交。

## Git 上傳前確認規則

執行 `git push` 或設定 / 修改 remote 前，必須先確認目前專案與目標 Git repo 對得上，避免把檔案推到錯誤 repo。

push 前必須檢查：

- 目前工作目錄路徑與資料夾名稱。
- `git remote -v` 的目標 URL；若尚未設定 remote，則檢查準備要設定的 repo URL。
- 目前 branch 與 upstream，例如 `git status --short --branch`；若尚未設定 upstream，則確認準備推送的 branch。
- 最新 commit 內容，例如 `git log -1 --oneline`。
- 將要提交或推送的檔案清單，例如 `git status --short`、`git diff --stat`、`git diff --cached --stat`。

本機資料夾名稱、README/專案名稱、remote repo 名稱與 branch 不要求完全相同，但必須能合理對應到同一個專案。若出現無法合理對應的差異、看起來像不同專案、或與使用者明確指定的目標不符，必須先告知使用者並停止 push，等使用者再次確認後才能繼續。

若 remote 已存在但看起來指向不同專案，不得自行覆蓋或改 remote；必須先回報目前 remote 與預期目標的差異。

## Agent 進度記錄規則

每次開始工作前，agent 必須先閱讀：

- `AGENTS.md`
- `AGENT_PROGRESS.md`（如果存在）
- `README.md`（如果存在）

`AGENT_PROGRESS.md` 用來記錄跨 agent 交接所需的目前進度與重點討論內容。不要建立 `docs/DECISIONS.md`；目前所有交接重點集中記錄在 `AGENT_PROGRESS.md`。

工作過程中若出現以下情況，必須主動更新 `AGENT_PROGRESS.md`：

- 使用者做出重要決定。
- 需求、方向或優先順序改變。
- 完成一個明確階段。
- 修改重要檔案、資料夾結構或專案規則。
- 遇到阻礙、錯誤或待確認事項。
- 任務暫停、對話即將結束，或準備交接給下一個 agent。

`AGENT_PROGRESS.md` 應維持精簡，只記錄交接需要知道的資訊，不寫機密資訊、不寫冗長聊天逐字稿。

任務收尾前，agent 必須檢查：

- `AGENT_PROGRESS.md` 是否需要更新。
- 若有環境變數變更，`.env` 與 `.env.example` 是否同步。
- 若有使用方式、資料夾用途或重要規則變更，`README.md` 是否需要更新。
- 若有測試，是否已執行；若無法執行，必須在回覆中說明原因。
